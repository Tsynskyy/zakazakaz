## 3. Домены

| Домен        | Ответственность                                | Данные                                     |
| ------------ | ---------------------------------------------- | ------------------------------------------ |
| User         | Регистрация, аутентификация, профили, роли     | users, roles, auth_tokens                  |
| Catalog      | CRUD товаров, категории, поиск                 | products, categories, stock                |
| Order        | Жизненный цикл заказа, смена статусов          | orders, order_items, order_status          |
| Payment      | Расчёт стоимости, проведение платежа, возвраты | payments, transactions, refunds            |
| Feed         | Персонализированная лента                      | user_preferences, view_history, feed_cache |
| Notification | Отправка email/push/SMS по событиям            | notification_templates, delivery_log       |

## 4. Домены -> Сервис

| Сервис               | Домен        | Почему отдельно                                         |
| -------------------- | ------------ | ------------------------------------------------------- |
| User Service         | User         | Изолированная область с собственной моделью данных      |
| Catalog Service      | Catalog      | Основная нагрузка чтения, независимое масштабирование   |
| Order Service        | Order        | Сложный жизненный цикл, много смен статусов             |
| Payment Service      | Payment      | Финансовые данные, изоляция                             |
| Feed Service         | Feed         | Другой профиль нагрузки (чтение + кэш), готовность к ML |
| Notification Service | Notification | Слушает события из брокера, отправляет уведомления      |

## 5. Границы данных и взаимодействия

Отсутствуют разделяемые базы данных между сервисами, каждый сервис владеет своей БД

Основной тип связи - Domain coupling через стабильные контракты. Pass-through, Common и Content coupling отсутствуют (нет общих БД, нет циклов, нет отправки/получения данных через промежуточные сервисы)

Синхронно: Gateway -> User/Catalog/Order/Feed, Order -> Catalog (проверка и резерв остатков), Order -> Payment (оплата), Feed -> Catalog (товары), Feed -> User (предпочтения)

Асинхронно: Order -> Broker (`order_created`, `order_status_changed`), Payment -> Broker (`payment_succeeded`, `payment_failed`), Broker -> Notification

## 6. Альтернативные варианты

Вариант 6 микросервисов: каждый домен - сервис

- Плюсы: независимый деплой и масштабирование, Payment изолирован, соответствие закону Конвея
- Минусы: больше инфраструктуры (6 БД, мониторинг), задержка межсервисных вызовов

Вариант 3 сервиса: UserCatalogFeed + OrderPayment + Notification

- Плюсы: меньше сетевых вызовов, проще согласованность данных, быстрый старт
- Минусы: Feed/Catalog нельзя масштабировать раздельно, Payment не изолирован, деплой затрагивает несколько доменов

## 7. Trade-off'ы

| Критерий             | 6 сервисов                              | 3 сервиса                          |
| -------------------- | --------------------------------------- | ---------------------------------- |
| Масштабирование      | Независимое                             | Только целиком сервис из 3 доменов |
| Изоляция Payment     | Отдельный контур                        | В одном процессе с Order           |
| Задержки             | Выше                                    | Ниже                               |
| Транзакции           | Распределённые, сложнее согласованность | Проще согласованность              |
| Инфраструктура       | Больше                                  | Меньше                             |
| Независимость деплоя | Полная                                  | Ограниченная                       |

## 8. Обоснование выбора

Выбранный вариант - 6 сервисов

- Feed и Catalog имеют разный профиль нагрузки: Feed - много чтения и кэширования, Catalog - частая запись от продавцов. В одном сервисе их нельзя масштабировать независимо
- Payment работает с финансовыми данными - его удобно отделить от других
- Feed меняется часто (эксперименты с рекомендациями), User и Order меньше - разные циклы деплоя
- 6 доменов могут быть на 6 команд (закон Конвея)
- Внутри каждого сервиса все относится к одному домену (высокий cohesion), между сервисами - только доменная связанность (низкий coupling)
